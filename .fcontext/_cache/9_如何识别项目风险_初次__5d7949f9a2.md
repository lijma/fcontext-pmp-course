# 最终要达到的效果
03
| 风险类别 | 风险点描述/举例 | 在哪里看 | 备注 |
| --- | --- | --- | --- |
| 硬件风险 | 1、存储不足 2、服务器型号低导致计算能力不足（并发不足） | 《建设方案》 《合同》 | 总则：发现的这些风险要找个小本本记下来，开交底会的时候，把《建设方案》、《投标文件》、《合同文件》中的内容作为证据一个个过。统一认知、明确职责、防止扯皮 |
| 架构风险 | 1、涉及的网络框架是否存在延时的情况（常见于多级部署架构） 2、架构不合理（小概率，能看出来的PM绝逼大神） | 《建设方案》 |  |
| 合同风险 | 1、合同约定的进度，按照当前资源估算无法满足 2、验收标准不明确 备注：更多内容详见《6. 如何识别项目范围》章节讲的如何看合同 | 《合同》 | 1、加人能解决，就和销售一起找公司要资源 2、加人不能解决：和销售一起去搞客户 |
| 客户风险 | 1、抢来的合同，客户大概率会不配合； 2、客户期望值过高 3、客户内部不和 | 销售 | 目的在于要让销售在关键时刻去攻关，不让他们做甩手掌柜 |
| 产品风险 | 1、产品一坨、功能缺陷 2、产品价值本身就不明显 | 自己评估 | 也只能是说说，让销售心里有底，后期不会怪我们 |
| 人员风险 | 1、人不够 2、能力不行 | 自己评估 | 说了也没用，销售或领导会认为是人力储备或培养没做好，自己心里清楚就行了 |
| 成本风险 | 利润率过低或不盈利 | 综合评估 | 根据上述文档、销售给的信息综合评估 |
